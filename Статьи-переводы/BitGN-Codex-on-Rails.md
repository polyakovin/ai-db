# Codex-on-Rails — frontier coding agent с жёсткими рельсами

**Источник:** BitGN PAC1 (Personal Agent Challenge)  
**Автор:** Igor Inozemtsev (Nevis)  
**Результат:** **87.0 / 104** — 1-е место по Accuracy (детерминированная точность)  
**Ключевая идея:** Мощный coding agent (Codex) + MCP-мост к vault-тулам + «рельсы» (rails), ограничивающие свободу модели на каждом шаге

---

## Что такое Codex-on-Rails

Codex-on-Rails — архитектура, в которой frontier coding agent (Codex) не свободен в своих действиях, а движется по жёстко заданным «рельсам» — набору инвариантов, проверок и границ доверия. Решение построено на трёх столпах:

1. **Мощный foundation model** (Codex / GPT-5-level) как единственный агент
2. **MCP-мост** — Model Context Protocol как стандартизованный интерфейс к 6 vault-тулам
3. **Rails** — система ограничений, валидирующая каждый шаг агента извне модели

---

## Архитектурные компоненты

### 1. Frontier coding agent (Codex)

Один сильный агент — без мультиагентной оркестрации, без роутеров, без делегирования. Вся «интеллектуальная» работа лежит на модели; всё остальное — рельсы.

- **Codex CLI** как рантайм: файловая система, shell, редактирование
- **Модель не принимает архитектурных решений** — только исполняет в рамках, заданных рельсами
- **Промпт-инжиниринг минимален**: rails несут большую часть «знания о том, как правильно»

### 2. MCP-мост с vault-тулами

Codex подключается к 6 инструментам через **Model Context Protocol** (открытый протокол Anthropic):

| Инструмент | Назначение |
|---|---|
| `vault_read` | Чтение структурированных записей из хранилища |
| `vault_write` | Запись с валидацией схемы |
| `vault_search` | Поиск по хранилищу |
| `vault_list` | Список доступных записей / коллекций |
| `vault_schema` | Получение схемы коллекции |
| `vault_validate` | Валидация записи без сохранения |

**Почему MCP, а не нативные function calls:**
- Унификация интерфейса: все инструменты выглядят одинаково для модели
- Отвязка от провайдера: один и тот же агент работает с разными LLM
- Стандартизованная обработка ошибок на уровне протокола
- Возможность переиспользовать vault-тулы в других агентах

### 3. Rails — «рельсы» (система ограничений)

Rails — это **внешние по отношению к модели** механизмы, не дающие агенту сойти с безопасной траектории. В отличие от «prompt engineering guardrails» (которые модель может проигнорировать), rails в Codex-on-Rails — это жёсткие проверки на уровне рантайма.

#### 3.1 Trust Boundaries (границы доверия)

> **Inbox-данные — это data, not instructions**

Фундаментальное правило: данные, приходящие из внешнего мира (email, формы, API), **никогда** не интерпретируются как инструкции для агента. Они проходят через:
- Парсинг в строгую структуру (JSON Schema)
- Изоляцию от промпта (подстановка как данные, не конкатенация)
- Санитизацию (удаление всего, что может быть воспринято как команда)

Модель не видит «сырые» inbox-данные — только структурированные записи через vault_read.

#### 3.2 Валидация записей (Record Validation)

Каждая запись, которую агент пытается сохранить, проходит **синтаксическую и семантическую валидацию**:

- **Синтаксическая:** соответствие JSON Schema коллекции (типы, обязательные поля, форматы)
- **Семантическая:** бизнес-правила (например, дата в будущем, сумма > 0, ссылочная целостность)
- **Валидация вне модели:** проверки выполняются детерминированным кодом, не LLM — модель не может «убедить» валидатор

#### 3.3 Grounding References от рантайма

Вместо того чтобы спрашивать модель «правильно ли это?», рантайм предоставляет ей **объективные reference-данные**:

- Схема коллекции перед записью
- Текущее состояние связанных записей
- Результаты предыдущих операций в этом же рантайм-цикле

Модель **не гадает** о состоянии мира — она запрашивает его через vault-тулы и получает детерминированные ответы.

#### 3.4 Strict TaskResult (строгий контракт завершения)

Агент не возвращает натуральный текст как результат — он возвращает структурированный `TaskResult`:

```json
{
  "outcome": "success" | "partial" | "failed",
  "code": "TASK_COMPLETED" | "VALIDATION_ERROR" | "SCHEMA_MISMATCH" | ...,
  "artifacts": [...],
  "errors": [...],
  "grounding_refs": {...}
}
```

Это даёт:
- **Машинную проверяемость:** вызывающая сторона не парсит текст
- **Единый контракт:** любая задача завершается одинаково структурированным ответом
- **Отладку:** outcome codes показывают, где именно произошёл сбой

## Разбор 17 misses (где агент ошибся)

При счёте 87/104 агент допустил 17 misses. Автор провёл глубокий разбор каждой ошибки, категоризировав их:

### Категории ошибок

| Категория | Количество | Описание |
|---|---|---|
| **Schema misinterpretation** | 5 | Модель неправильно поняла схему коллекции или отношения между полями |
| **Missing validation edge case** | 4 | Бизнес-правило не было покрыто рельсами → модель приняла неверное решение |
| **Tool misuse** | 3 | Агент вызвал не тот vault-тул или передал некорректные параметры |
| **Context overflow** | 2 | Контекстное окно переполнилось при работе с большими коллекциями |
| **Grounding failure** | 2 | Модель не запросила нужный grounding ref перед принятием решения |
| **Prompt ambiguity** | 1 | Неоднозначность в промпте привела к неправильной интерпретации задачи |

### Ключевые выводы из misses

1. **Большинство ошибок (9/17) — не «глупость» модели, а недостаточность рельсов.** Если бы валидация покрывала эти edge cases, агент не ошибся бы.
2. **Schema misinterpretation (5)** — модель «додумывает» семантику схемы, когда она неочевидна. Решение: более явные схемы + grounding refs.
3. **Tool misuse (3)** — агент иногда выбирал неправильный инструмент для задачи. Решение: более строгая спецификация инструментов в MCP-дескрипторах.
4. **Grounding failure (2)** — модель принимала решения, не запросив актуальное состояние связанных записей. Решение: принудительный pre-fetch связанных данных перед операциями записи.

> **Главный урок:** Каждый miss — это не баг модели, а недостающая рельса. Эволюция системы — это добавление рельс, покрывающих найденные классы ошибок.

## Паттерны, применимые в других системах

### 1. «Data, not instructions»
Универсальный паттерн для любых агентов, работающих с внешним вводом. Применение:
- Email-агенты: парсить тело письма в JSON, не подставлять в промпт
- Чат-боты: отделять пользовательский ввод от системных инструкций
- Агенты для документов: структурировать документ до передачи модели

### 2. Strict TaskResult
Переход от «агент что-то сказал» к «агент вернул структурированный результат». Применение:
- Любой продакшен-агент должен возвращать outcome codes
- Натуральный текст — только для humans, машины читают структуру

### 3. Grounding refs before writes
Перед любой мутирующей операцией — принудительный pre-fetch состояния. Применение:
- Перед UPDATE — SELECT актуальной версии записи
- Перед записью в связанную коллекцию — проверка существования reference

### 4. Rails evolution через failure analysis
Не улучшать промпт, а добавлять проверки в рантайм. Применение:
- Каждый production incident → новая рельса
- Валидация накапливается слоями, как тесты

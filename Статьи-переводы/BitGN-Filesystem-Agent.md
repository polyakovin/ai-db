## Архитектурная идея

**Один сильный агент + классификатор сценариев + маленькие workflow-скиллы.** Вместо мультиагентного оркестратора — одна сильная модель (GPT-5.4), которая сама принимает решения, но опирается на лёгкие workflow-компоненты для специализированных доменов.

Ключевое отличие от [[BitGN-Operation-Pangolin|Pangolin]] (один `execute_code`) и [[BitGN-Codex-on-Rails|Codex-on-Rails]] (один coding agent + MCP-мост): Filesystem Agent **разделяет принятие решений и выполнение** через файловую систему как коммуникационную шину.

## Компоненты системы

### 1. Один сильный агент (GPT-5.4)
- Единственная точка принятия решений — одна модель, без мультиагентной координации
- Агент не делегирует подзадачи другим агентам, а **сам решает, какой workflow-скилл применить**
- ФС используется как рабочая память: читает файлы, пишет результаты, передаёт контекст между шагами

### 2. Классификатор сценариев (Scenario Classifier)
- Первый шаг: агент классифицирует входящий запрос по типу сценария
- Сценарии: `inbox` (обработка входящих), `security` (безопасность), `finance` (финансы), `documents` (работа с документами), `crm` и др.
- Классификатор — это не отдельная модель, а **промпт-этап** в рамках того же агента
- Результат классификации записывается в ФС и определяет, какой workflow-скилл активировать

### 3. Маленькие workflow-скиллы
- Каждый скилл — **лёгкий workflow-файл** в ФС (markdown или yaml), описывающий шаги для конкретного сценария
- Скиллы **не содержат логики** — только чеклист шагов и контракты данных
- Вся логика — в модели; скилл — это «шпаргалка», которую агент читает и исполняет
- Примеры скиллов:
  - `inbox.md` — классификация писем, извлечение action items, приоритизация
  - `security.md` — проверка подозрительных запросов, валидация доступа
  - `finance.md` — категоризация транзакций, сверка, отчёты

### 4. Evidence Tracking от рантайма
- Агент не полагается на «память» модели — все доказательства пишутся в ФС
- Каждый шаг workflow оставляет **evidence-файл**: что сделано, какой результат, какие данные использованы
- Валидация — на стороне рантайма: скрипты проверяют корректность выходных файлов **вне модели**
- Это реализует принцип **trust boundaries**: модель предлагает, рантайм проверяет

### 5. Inbox как data, not instructions
- Ключевой инсайт: входящие данные (письма, запросы, транзакции) — это **данные**, а не инструкции
- Никакой инжекции промптов через inbox: данные проходят через строгий парсинг и валидацию **до** того, как попадают в контекст модели
- Inbox-файлы хранятся в изолированной директории, модель читает их **после** предобработки
- Это реализует принцип **trust boundary** из [[BitGN-Arena-Insights|общих инсайтов PAC1]]: inbox-данные не могут менять поведение агента

## Почему это сработало

| Фактор | Механика |
|--------|----------|
| **Один сильный агент** | Нет накладных расходов на мультиагентную координацию; модель не путается в делегировании |
| **Классификатор сценариев** | Быстрый routing без дорогого RAG или эмбеддингов; промпт-инжиниринг вместо ML-пайплайна |
| **ФС как шина** | Данные не теряются между шагами; модель всегда может «оглянуться» и проверить контекст |
| **Маленькие скиллы** | Не перегружают контекст; каждый скилл — ~20–50 строк, а не мегапромпт |
| **Evidence tracking** | Аудируемость: всегда можно понять, почему агент принял решение |
| **Inbox = data** | Устранение целого класса уязвимостей (prompt injection через входные данные) |

## Связь с паттернами

Filesystem Agent реализует несколько ключевых паттернов:

- **[[Паттерны/Plan-and-Execute]]** — классификатор сценариев даёт план (какой скилл применить), а основной агент — execute. Но план не жёсткий: агент может отклониться, если evidence tracking показывает проблему.
- **[[Паттерны/Human-in-the-Loop]]** — evidence tracking создаёт «бумажный след», который человек может аудировать; критические решения (финансы, безопасность) могут требовать подтверждения.
- **Trust Boundaries** (см. [[BitGN-Arena-Insights]]) — inbox как data, валидация вне модели, изоляция сценариев.
- **Durable Memory через ФС** — аналог подхода из [[BitGN-Operation-Pangolin|Pangolin]], но вместо одного `memory.json` — структурированная файловая система с evidence-файлами.

## Проблемы и ограничения

- **Зависимость от качества классификатора** — если сценарий определён неверно, агент идёт по неправильному workflow
- **ФС как bottleneck** — много операций чтения/записи; на больших объёмах — латентность
- **Скиллы требуют ручного написания** — каждый новый сценарий = новый workflow-файл; не самообучается
- **Нет evolution-лупа** — в отличие от [[BitGN-Codex-CLI-Rules-Evolution|Codex CLI + Rules Evolution]], здесь нет автоматического улучшения правил на основе failure analysis
- **Один агент = одна точка отказа** — если модель hallucinate на уровне классификации, каскад ошибок неизбежен

## Вывод

Filesystem Agent — элегантный компромисс между минимализмом Pangolin (один инструмент) и структурированностью Codex-on-Rails (рельсы). Файловая система как коммуникационная шина + классификатор сценариев + evidence tracking — доказывает, что можно получить 83/104 **без сложной мультиагентной архитектуры**.

> *Ключевой инсайт: inbox — это data, not instructions. А файловая система — самый недооценённый инструмент агента.*

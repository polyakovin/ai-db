## Архитектурная идея

Агент построен вокруг **одного-единственного инструмента** — `execute_code`. Никаких вызовов API, file-тулов, search-тулов. Только песочница Python, в которой агент пишет, запускает и отлаживает код.

Это радикально минималистичный подход: вместо набора специализированных инструментов — универсальный REPL, где модель сама создаёт «инструменты» через код.

---

## Компоненты системы

### 1. `execute_code` — единственный инструмент
- Песочница Python с ограниченным доступом к сети и ФС
- Агент может читать/писать файлы, парсить, считать, вызывать API — **через код**
- Результат выполнения возвращается модели в виде stdout/stderr + exit code

### 2. REPL-цикл (Read-Eval-Print Loop)
- Агент пишет код → исполняет → читает вывод → пишет следующий код
- Каждый шаг — атомарный: один вызов `execute_code` = один осмысленный action
- Модель сама решает, когда цикл завершён и результат готов

### 3. Чеклист (Checklist)
- Структурированный список подзадач, который агент составляет перед работой
- Каждый пункт — проверяемое утверждение (done / not done)
- Чеклист живёт в `durable memory` и перечитывается на каждом шаге
- Агент **не завершает работу**, пока все пункты не отмечены `done`

### 4. Durable Memory
- Файловая «долговременная память» в песочнице: `memory.json` или аналог
- Сохраняет: чеклист, промежуточные результаты, обнаруженные факты
- При каждом шаге агент перечитывает memory — это заменяет дорогой context window
- Не полагается на «память» модели — всё явно сохраняется и загружается

### 5. Verify Gate (строгий гейт валидации)
- Финальный шаг перед ответом: агент **обязан** выполнить проверочный код
- Проверка синтаксическая (well-formed output) и семантическая (совпадение с ожидаемым контрактом)
- Если verify не проходит — агент возвращается в REPL-цикл
- Результат: **TaskResult с исходом**, а не свободный текст

## Почему это сработало

| Фактор | Механика |
|--------|----------|
| **Один инструмент** | Нет confusion от выбора tool — модель не тратит токены на tool-selection |
| **Рельсы через код** | Валидация и grounding — на стороне рантайма, не на совести модели |
| **Durable memory** | Нет деградации от длинного контекста — память вынесена в файлы |
| **Чеклист как контракт** | Модель сама себе ставит задачи и проверяет их выполнение |
| **Verify gate** | Жёсткий финальный барьер — «пока не проверил, не отдавай» |
| **REPL, а не Plan-then-Do** | Итеративное уточнение вместо хрупкого upfront-плана |

## Проблемы и ограничения

- **Латентность:** много раундов `execute_code` → больше шагов, чем у прямых вызовов API
- **Безопасность песочницы:** `execute_code` обязан быть жёстко изолирован (иначе дыра)
- **Cost:** каждый REPL-цикл — это токены на вход (код + вывод + memory + чеклист)
- **Отладка:** если код падает — модель может зациклиться; нужен таймаут
- **Не для всех задач:** подходит для задач, выразимых в коде; для «позвони по API» — overhead

## Вывод

Operation Pangolin — доказательство того, что **минимализм инструментов + жёсткие рельсы валидации** бьют сложные многоагентные архитектуры на строгом бенчмарке. Не мультиагент, не RAG, не цепочки тулов — один `execute_code` и дисциплина.

> *Ключевой инсайт: модель не должна выбирать инструменты. Модель должна писать код, который делает всё остальное.*

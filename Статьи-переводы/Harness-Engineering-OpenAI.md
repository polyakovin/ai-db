# Harness Engineering: Leveraging Codex in an Agent-First World

**Источник:** OpenAI Blog, 11 февраля 2026  
**Автор:** Ryan Lopopolo  
**Оригинал:** https://openai.com/index/harness-engineering/

## Ключевая идея

Построить и запустить программный продукт с **нулём строк кода, написанного вручную** — каждая строчка (тесты, CI, документация, инфраструктура) написана агентами Codex.

> «Люди направляют. Агенты исполняют.»

## Основные результаты

| Метрика | Значение |
|---|---|
| Timeline | ~5 месяцев — от пустого репозитория до internal beta + external alpha |
| Объём | ~1 млн строк кода, **1500 PR** |
| Скорость | 3.5 PR/инженер/день (команда 3→7 человек) |
| Роль человека | Ни строчки вручную — дизайн окружения, спецификация намерений, фидбек-лупы |
| Оценка | ~1/10 времени ручной разработки |

## Основные тезисы

### 1. Начало с нуля
- Первый коммит — август 2025, сгенерирован Codex CLI на GPT-5
- Даже `AGENTS.md` написан агентом
- Философия: «ни строчки вручную» — жёсткое ограничение

### 2. Новая роль инженера — «проводник агента»
- Не писать код, а **готовить задачи для агентов**:
  - Декомпозиция целей на мелкие legible блоки
  - Выявление недостающих возможностей → кодификация в репозитории
- Взаимодействие: описал задачу → запустил агента → агент открыл PR
- **Human review опционален** — основное ревью идёт агент→агент

### 3. Legibility приложения для агентов
- **Chrome DevTools Protocol** встроен в рантайм Codex → DOM-снапшоты, скриншоты, навигация
- **Observability-стек** (LogQL, PromQL, TraceQL) — агенты читают логи и метрики напрямую
- Агенты регулярно выполняют задачи по **6+ часов** (часто — ночью)

### 4. Знания репозитория — system of record
- **Один большой `AGENTS.md` не работает**: контекст вытесняет задачу, всё labelled «важно» → шум, ротация устаревает
- Решение: `AGENTS.md` как **оглавление (~100 строк)** + структурированная `docs/`:
  - design-docs/, exec-plans/ (с active, completed, tech-debt-tracker), generated/, product-specs/, references/
  - Планы — first-class артефакты (execution plans с логами прогресса, зачекинины в репозиторий)
  - Линтеры/CI валидируют свежесть знаний
  - **«Doc-gardening» агент** — ищет устаревшие доки, открывает fix-up PR

### 5. Всё в репозиторий
- Знания вне репозитория (Slack, Google Docs, головы людей) — невидимы агентам
- Каждое решение, архитектурный паттерн, норма команды — в markdown
- «Boring technologies» — композитные, стабильные, хорошо представлены в тренировочных данных
- Иногда выгоднее **переписать маленький модуль**, чем бороться с непрозрачной зависимостью

### 6. Архитектура и вкус через код
- **Custom линтеры** (написаны Codex) enforcing:
  - Строгая layered-архитектура (Types → Config → Repo → Service → Runtime → UI)
  - Cross-cutting concerns — только через explicit `Providers` interface
  - Structured logging, naming conventions, лимиты на размер файла
- **Error messages для контекста агента** — включают remediation instructions
- Фидбек: ревью-комментарии → доки → encoded в тулинг

### 7. Merge-философия
- Минимальные blocking gates — PR живут недолго
- Flaky тесты — fix через follow-up, не блокировка
- **Corrections cheap; waiting expensive**

### 8. Что значит «сгенерировано агентом»
Всё — продукты, тесты, CI, релиз-тулинг, devtools, docs, evaluation harnesses, review comments, скрипты управления репозиторием, dashboard definition files.

### 9. Рост автономии
По одному промпту Codex может:
1. Проверить текущий кодбазу на архитектурные нарушения
2. Найти точки отказа по метрикам
3. Открыть PR с правками и читаемыми коммитами

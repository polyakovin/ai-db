## Что такое Plan-REPL Agent

Plan-REPL Agent — агентная архитектура, построенная вокруг жёсткого разделения ролей между моделями и фиксированного цикла «спланировал → исполнил → решил, что дальше». Главная ставка сделана на **качество промптов**, а не на сложность рантайм-проверок или количество инструментов.

Архитектура состоит из четырёх фаз, выполняемых последовательно:

1. **Preflight check** — быстрая валидация входа, определение типа задачи
2. **Planning (GPT-5.4)** — сильная модель строит пошаговый план
3. **Step execution (GLM-5.1)** — более лёгкая/дешёвая модель исполняет шаги в REPL-стиле
4. **After-step decision** — анализ результата шага и выбор: продолжать, корректировать или завершать

---

## Архитектурные компоненты

### 1. Preflight check — валидация до старта

Перед тем как запускать дорогую модель-планировщик, агент выполняет быструю проверку:

- **Валидация входных данных:** проверка формата, полноты, отсутствия явных аномалий
- **Классификация задачи:** определение типа сценария (CRUD, аналитика, поиск, агрегация)
- **Оценка сложности:** нужно ли планирование или задача решается одним вызовом?
- **Подготовка контекста:** структурирование входных данных для планировщика

Preflight — это лёгкий детерминированный (или почти детерминированный) слой, не требующий вызова сильной LLM.

### 2. Planning — GPT-5.4 как «мозг»

Ключевая ставка архитектуры: **сильная модель только планирует, не исполняет**.

- **Модель:** GPT-5.4 (frontier-level reasoning)
- **Вход:** структурированный контекст от preflight
- **Выход:** пошаговый план — typed step outputs (каждый шаг имеет тип и ожидаемую структуру результата)
- **Промпт-инжиниринг — ядро фазы:** план строится исключительно качеством промпта: явные инструкции, few-shot примеры, описание форматов инструментов

Планировщик **не имеет доступа к инструментам** во время планирования — он только проектирует последовательность шагов, которые будет выполнять другая модель.

### 3. Step execution — GLM-5.1 как «руки»

Каждый шаг плана исполняется отдельным вызовом более лёгкой модели:

- **Модель:** GLM-5.1 (быстрее и дешевле GPT-5.4, хороший баланс tool-use/стоимость)
- **Вход:** описание текущего шага из плана + результаты предыдущих шагов + доступные инструменты
- **Выход:** typed step output — структурированный результат в формате, заданном планировщиком
- **REPL-стиль:** исполнитель работает в цикле «прочитал состояние → вызвал инструмент → получил результат → записал вывод»

Мотивация разделения:
- GPT-5.4 дорогая — использовать её на каждом шагу нерентабельно
- GLM-5.1 достаточно умна для изолированного шага, но ей не нужно «понимать» всю задачу целиком
- План служит «рельсами» для исполнителя — он не придумывает стратегию, а следует готовому плану

### 4. After-step decision — что делать дальше

После каждого исполненного шага агент принимает решение:

- **Продолжить:** шаг выполнен успешно → следующий шаг по плану
- **Перепланировать:** результат шага не соответствует ожиданиям → возврат к планировщику для корректировки плана
- **Завершить:** все шаги выполнены → формирование итогового ответа

Решение принимается **тем же исполнителем** (GLM-5.1) или лёгким детерминированным правилом — без вызова тяжёлой модели. Это создаёт риск: если исполнитель неверно оценит свой результат, ошибка каскадно пройдёт дальше.

## Почему 60.6/104: разбор ограничений

При счёте 60.6/104 агент показал значительно более низкий результат, чем топ-решения (Pangolin — 92.0, Codex-on-Rails — 87.0). Основная причина — **лимиты чистого prompt engineering при фиксированной архитектуре**.

### Ключевые слабые места

| Ограничение | Проявление | Контраст с топ-решениями |
|---|---|---|
| **Нет внешней валидации** | Модели доверяют своим же ответам; нет rails/Grounding refs | Codex-on-Rails: каждая запись валидируется вне модели |
| **Разрыв план-исполнение** | Исполнитель неправильно интерпретирует план, нет жёсткого контракта между фазами | Pangolin: один агент в REPL-цикле, нет разрыва |
| **Промпт вместо рельс** | Корректность держится на качестве промптов, а не на рантайм-проверках | Codex-on-Rails: рельсы эволюционируют независимо от промпта |
| **After-step decision без grounding** | Решение «продолжить или перепланировать» принимается на основе вывода модели, а не объективного состояния | Codex-on-Rails: Strict TaskResult с outcome codes |
| **Две модели — два источника ошибок** | Каждая модель может ошибиться по-своему; ошибки умножаются | Pangolin, Codex-on-Rails: одна модель |
| **Нет эволюционного лупа** | Если агент ошибается, фикс — править промпт; нет механизма накопления «рельс» | Codex CLI Rules Evo: отдельный evolution-луп |

### Главный урок

> **Plan-REPL Agent показал потолок подхода «всё решается промптами».** Как бы хороши ни были промпты планировщика и исполнителя, без внешней валидации, без жёстких контрактов между фазами, без эволюционного накопления проверок — агент упирается в ~60% точности на PAC1. Дальнейший рост требует архитектурных изменений, а не улучшения промптов.

| **Trust boundary** | Отсутствует (модель видит сырой ввод) | Встроен в execute | Data, not instructions | AGENTS.md |

## Паттерны, применимые в других системах

### 1. Preflight check перед дорогими вызовами
Лёгкая валидация и классификация задачи **до** вызова сильной LLM — универсально полезный паттерн. Экономит токены и время, отсеивая тривиальные или некорректные запросы на раннем этапе.

### 2. typed step outputs
Структурирование выхода каждого шага с явным типом — полезная практика даже в одно-модельных архитектурах. Облегчает отладку и позволяет детерминированно проверять результаты.

### 3. Разделение «думать» и «делать» (с оговорками)
Идея использовать сильную модель для планирования и более лёгкую для исполнения — привлекательна с точки зрения стоимости. Однако Plan-REPL показал, что **без жёсткого контракта между фазами** это разделение создаёт больше проблем, чем решает.

### 4. After-step gating
Проверка результата каждого шага перед продолжением — необходимый паттерн. Но он должен опираться на **объективные критерии** (outcome codes, schema validation), а не на суждение той же модели.

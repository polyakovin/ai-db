# Таксономия AI-агентов

Термин “агент” часто используют слишком широко. Для проектирования полезнее разделять системы по уровню автономности, наличию tools, состоянию и ответственности за side effects.

## Основные типы

| Тип | Что делает | Есть tools | Есть автономный loop | Типичный пример |
|---|---|---:|---:|---|
| Assistant | Отвечает на запрос пользователя | Иногда | Нет | чат с документами |
| Workflow | Выполняет заранее заданный граф шагов | Да | Нет или ограниченно | классификация заявки -> генерация ответа -> CRM |
| Agent | Сам выбирает следующие действия внутри заданной цели | Да | Да | research agent, coding agent |
| Autonomous agent | Работает долго, сам планирует, проверяет и продолжает | Да | Да | агент мониторинга, backlog triage |
| Multi-agent system | Несколько агентов с разными ролями и общим протоколом | Да | Да | supervisor + researcher + reviewer |

## Уровни автономности

| Уровень | Описание | Контроль человека |
|---|---|---|
| L0 | Модель только отвечает текстом | пользователь вручную действует сам |
| L1 | Модель предлагает действие, но не выполняет | confirmation перед каждым действием |
| L2 | Модель выполняет безопасные tools | approval для рискованных side effects |
| L3 | Модель планирует и выполняет многошаговую задачу | review gates и лимиты |
| L4 | Модель работает фоново по событию или расписанию | policy, audit, incident response |
| L5 | Широкая автономия в открытой среде | для большинства продуктов это нежелательно |

## Когда агент не нужен

Агентность добавляет стоимость, latency, риск и сложность отладки. Не нужен агент, если:

- задача решается одним deterministic workflow;
- нет необходимости выбирать tools динамически;
- все шаги известны заранее и редко меняются;
- цена ошибки высока, а human approval всё равно требуется на каждом шаге;
- retrieval + structured output покрывают сценарий без planning loop.

В таких случаях лучше построить pipeline, обычный assistant или workflow engine.

## Признаки настоящей agentic-задачи

Агент оправдан, если:

- нужно исследовать неизвестное состояние среды;
- путь к результату зависит от observation после каждого шага;
- tools имеют разные failure modes;
- требуется восстановление после ошибок;
- задача длиннее одного prompt-response;
- полезно сохранять state, evidence и промежуточные решения;
- пользователь задаёт outcome, а не точный алгоритм.

## Границы ответственности

Модель не должна быть единственным местом, где живут правила. Разделение такое:

- модель выбирает действие и формулирует аргументы;
- harness валидирует permissions, schema, policy и stop criteria;
- tools выполняют ограниченные операции;
- evals проверяют качество;
- observability объясняет, что произошло;
- человек подтверждает рискованные решения.

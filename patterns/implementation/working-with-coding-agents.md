# Работа с код-агентами

Код-агент эффективен не тогда, когда "сам всё делает", а когда получает ясную цель, границы, проверяемый результат и доступ к правильным инструментам. Из ресурсов по [Claude Code](../../tools/platforms/anthropic.md), Codex, Superpowers, ECC и Karpathy-guidelines складывается практический workflow ниже.

## Базовый цикл

1. **Уточнить цель:** что должно измениться и как понять, что задача закрыта.
2. **Зафиксировать ограничения:** какие файлы, стили, API, сроки и риски важны.
3. **Сделать план:** короткие шаги, каждый со способом проверки.
4. **Выполнить маленькими порциями:** не смешивать исследование, реализацию и рефакторинг.
5. **Проверить:** тесты, линтеры, ручной сценарий, link-checker или другой релевантный gate.
6. **Сохранить след:** commit, summary, ссылки на изменённые файлы, известные ограничения.

## Чем цель лучше команды

Слабая постановка: “добавь валидацию”.

Сильная постановка: “некорректный email должен возвращать ошибку, существующие валидные сценарии не должны сломаться; добавь тест на invalid input и запусти suite”.

Агентам проще автономно доводить работу до конца, когда им заданы критерии успеха, а не только действие.

## Когда нужно остановиться и спросить

Агент должен не продолжать молча, если:

- есть несколько несовместимых трактовок задачи;
- нужный источник истины отсутствует;
- команда просит потенциально destructive action;
- изменение затрагивает чужие незавершённые правки;
- проверка невозможна без внешнего доступа или секрета;
- лучший вариант отличается от буквальной просьбы пользователя.

## Практика минимального diff

Хороший код-агент:

- не форматирует соседний код без причины;
- не меняет комментарии, которые не понимает;
- не добавляет “универсальность” без запроса;
- не удаляет старый dead code, если не он его создал;
- повторяет стиль проекта;
- делает refactor только если он нужен для задачи.

Это напрямую снижает стоимость ревью.

## Практика verification loop

Для каждой задачи заранее выбрать проверку:

| Тип задачи | Проверка |
|---|---|
| Bugfix | тест, который воспроизводит баг, затем проходит |
| Markdown/vault | link-checker, `git diff --check`, обзор ссылок |
| UI | локальный сервер, browser screenshot, responsive smoke |
| Refactor | тесты до/после, отсутствие изменения поведения |
| Инфраструктура | dry-run или минимальная команда, которая повторяет CI path |
| Документация | проверка ссылок, соответствие README/overview/TODO |

## Практика research-first

Если задача зависит от внешних фактов, версий моделей, документации API или актуальных курсов, агент должен сначала проверить источники. Хороший research-first workflow:

1. Найти первичные источники.
2. Зафиксировать дату актуальности.
3. Отделить факты от выводов.
4. Сохранить ссылки в заметке.
5. Не притворяться, что динамическая информация стабильна.

## Практика project memory

Повторяющиеся предпочтения пользователя должны становиться проектными правилами, а не оставаться в чате. Для этого подходят:

- `.nessy/memory/` — локальная память проекта;
- `README.md` — публичные правила для участников;
- `Skills/` — операционные процедуры;
- `AGENTS.md` или аналог — если проект использует конкретный agent harness.

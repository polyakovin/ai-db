# Skills и правила для агентов

Skills — это компактные, переиспользуемые инструкции или процедуры, которые агент подгружает под конкретную задачу. В отличие от большого монолитного system prompt, skills позволяют держать базовое поведение коротким, а специализированные практики включать только когда они нужны.

## Проблема

Большой набор правил в одном prompt быстро становится шумом:

- модель перестаёт понимать, что важно именно сейчас;
- контекст забивается редко используемыми инструкциями;
- правила устаревают и конфликтуют;
- команды начинают “улучшать” prompt вместо того, чтобы проверять поведение.

Skills решают это через модульность: каждая процедура имеет название, триггеры, шаги, критерии завершения и связанные проверки.

## Что должно быть в skill

Хороший skill отвечает на пять вопросов:

1. Когда его применять.
2. Какие входные данные нужны.
3. Какие шаги выполнить.
4. Как проверить результат.
5. Какие ошибки и ограничения учитывать.

Минимальная структура:

```markdown
# Название

## Когда использовать

## Алгоритм

## Проверка результата

## Частые ошибки

## Связанные материалы
```

## Полезные категории skills

| Категория | Примеры |
|---|---|
| Planning | brainstorming, writing-plans, decomposition |
| Implementation | TDD, executing-plans, framework-specific patterns |
| Debugging | systematic-debugging, root-cause tracing, verification-before-completion |
| Review | requesting-code-review, receiving-code-review |
| Knowledge | add-source, documentation lookup, research-first workflow |
| Safety | sandbox checks, secret handling, untrusted input boundaries |
| Operations | git workflow, status reports, release checklist |

## Принципы из Karpathy-inspired guidelines

Эти правила особенно полезны как базовые agent rules:

- **Think before coding:** явно назвать предположения, неоднозначности и tradeoffs.
- **Simplicity first:** не добавлять speculative flexibility и abstractions “на будущее”.
- **Surgical changes:** менять только то, что связано с задачей.
- **Goal-driven execution:** превращать команды в проверяемые критерии успеха.

В проектных правилах это лучше записывать не как моральные пожелания, а как наблюдаемые требования: “каждая изменённая строка должна быть связана с задачей”, “после изменения запустить X”, “если есть неоднозначность Y — спросить”.

## Правило: skills должны быть тестируемыми

Superpowers и ECC сходятся в одной идее: полезность skill нужно проверять поведением агента. Признаки, что skill работает:

- меньше лишних изменений в diff;
- меньше повторных переделок из-за переусложнения;
- агент спрашивает до ошибки, а не после;
- результат проходит проверки без ручного “доталкивания”;
- workflow можно повторить в другом проекте.

## Когда skill вреден

Skill стоит переписать или удалить, если он:

- слишком общий и повторяет базовые инструкции;
- не имеет явных триггеров;
- заставляет агента делать лишние шаги для простой задачи;
- конфликтует с проектными правилами;
- содержит устаревшие команды или пути;
- не объясняет, как проверить результат.

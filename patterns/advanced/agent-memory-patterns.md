# Персистентная память агента

Как агент сохраняет контекст, предпочтения и решения между сессиями и задачами. Без явной памяти каждый запуск начинается с нуля, а длинные multi-step задачи упираются в лимит context window.

## Проблема

- Контекстное окно конечно — агент «забывает» ранние шаги в длинных задачах
- Пользовательские предпочтения теряются между сессиями
- Повторяющиеся ошибки не фиксируются
- Невозможно восстановить ход рассуждений после сбоя

## Типы памяти

| Тип | Хранилище | Жизненный цикл | Пример |
|-----|-----------|----------------|--------|
| Session memory | Context window | Сессия | Текущий диалог, временные результаты |
| Working memory | Filesystem / notes | Задача | План, evidence, статусы шагов |
| Project memory | Markdown-файлы (MEMORY.md, AGENTS.md) | Проект | Правила, конвенции, решения, user preferences |
| Persistent memory | SQLite / векторная БД | Бессрочно | Чекпоинты, сессионные сводки, паттерны |

## Паттерны реализации

### 1. Filesystem как внешняя память

Файловая система — самое надёжное хранилище для агента. Хороший паттерн:

- `MEMORY.md` — долговременные факты о проекте
- `NOTES.md` или `scratch/` — временные заметки и evidence
- `plans/` — чекпоинты выполнения
- Каждый файл — явный артефакт, видимый в diff

### 2. Checkpointing

Периодическое сохранение состояния задачи позволяет:
- Восстановиться после сбоя или переполнения контекста
- Откатиться до предыдущего чекпоинта при ошибке
- Аудировать принятые решения

Агенты вроде [MiMo Code](../../tools/models/mimo-code.md) используют checkpoint-writer subagent, который сохраняет состояние параллельно основной работе.

### 3. Session distillation

После завершения задачи агент сжимает сессию в структурированную сводку:
- Что было сделано
- Какие решения приняты
- Какие ограничения обнаружены
- Что стоит запомнить для будущих задач

Этот паттерн используется в `/dream` команде [MiMo Code](../../tools/models/mimo-code.md) (самообучение каждые ~7 дней).

### 4. Embedding-based retrieval

Для баз знаний и RAG-сценариев память можно организовать через векторный поиск:
- Чанки знаний → embeddings → векторная БД
- Запрос агента → retrieval топ-K релевантных фактов
- Инъекция в контекст как grounding

## Когда какая память нужна

| Сценарий | Тип памяти |
|----------|-----------|
| Чат без сохранения | Session memory только |
| Код-агент, правящий репозиторий | Filesystem + Project memory |
| Research-агент | Session + Filesystem + Embedding-based |
| Multi-day coding проект | Все 4 уровня |
| Production-агент с аудитом | Persistent checkpointing + logs |
